# Claude Code Skills

## What This Repo Is

A test-bed for developing, improving, and deploying Claude Code skills. Skills are prompt-based plugins — markdown files with YAML frontmatter that Claude loads and follows as playbooks. Skills are grouped by workflow family at the repo root: each top-level directory (`milestone-driven/`, `research/`, `codebase-survey/`, `common/`) contains its own `skills/` and `agents/` subdirectories. A skill therefore lives at `<workflow>/skills/<skill-name>/SKILL.md` and a custom subagent at `<workflow>/agents/<name>.md`.

When working in this repo, the goal is typically to iterate on skill prompts, test them against real usage, and prepare them for deployment to other projects via `install.sh` (symlinks) or manual copy.

**Where testing happens:** This repo only contains skill and agent source — there are no skill executions, test runs, or generated artifacts here. When debugging or analyzing a skill or agent's behavior, the actual invocations (logs, outputs, generated files) live in a separate project where the skill was installed and run. Don't search this repo for test results, example outputs, or runtime traces; ask the user which project to look in instead.

## Installation

```bash
# Install every workflow globally (~/.claude/skills/ + ~/.claude/agents/ + ~/.claude/workflows/)
./install.sh all

# Install one workflow globally
./install.sh milestone-driven

# Install one workflow into a specific project
./install.sh research /path/to/project
```

`install.sh` takes an optional `--copy` flag, then the workflow name (or `all`), then an optional target directory. All selected skills are symlinked into `<target>/.claude/skills/`, agents into `<target>/.claude/agents/`, and any Workflow scripts (`<workflow>/workflows/*.js`) into `<target>/.claude/workflows/` — the flat layout Claude Code expects — so workflow grouping exists only at the source. Passing `--copy` copies the files instead of symlinking them, vendoring a self-contained snapshot into the target that is decoupled from this repo (useful for checking the tools into a project tree). Two workflows that ship a skill or agent with the same filename will shadow each other when installed together; this is intentional, so a workflow can re-define a skill by name when installed alone.

## Skill Architecture

### Workflow Families

**Milestone-driven workflow** — a four-phase cycle for building software:
1. `/project-inception` → Socratic dialogue producing README.md (one-time, precedes the cycle)
2. `/strategic-planning` → Socratic dialogue adding milestones (each as `roadmap/NNNN-slug.md`, indexed by one line in ROADMAP.md)
3. `/milestone-breakdown` → Decomposes a milestone into ordered tasks in PLAN.md; delegates codebase reconnaissance to the `milestone-scout` subagent
4. `/task-implementation` → Strict TDD: one task per invocation, tests first
5. `/implementation-cycle` → Sequentially spawns one `task-worker` subagent per task (which invokes task-implementation + commit) to keep the main session clean; after each task lands, spawns a `doc-updater` subagent to sync reference docs and examples to that one commit (no-op unless the change is surface-visible)
6. `/milestone-closing` → Verifies criteria, documents results (holistic README narrative pass), resets PLAN.md
7. `/implementation-cycle-run` → **Experimental** workflow-backed twin of `/implementation-cycle`: a standalone Workflow script (`milestone-driven/workflows/implementation-cycle.js`, `meta.name: implementation-cycle-run`) that drives the whole per-task loop as a scripted `agent()` fan-out instead of an in-session prose loop, so the two orchestration styles can be tested side by side. Same `task-worker` + `doc-updater` contract; because a Workflow script has no filesystem/bash access, it adds small `gate`/`verify` `general-purpose` subagents to stand in for the in-session git/PLAN.md reads

The milestone-driven workflow also ships one **Workflow script** under `milestone-driven/workflows/` (`implementation-cycle.js`, exposed directly as `/implementation-cycle-run`), installed by `install.sh` as a symlink into `<target>/.claude/workflows/` — the single-file-per-workflow layout mirrors `agents/`.

The milestone-driven workflow ships four custom subagents (`milestone-scout`, `task-worker`, `doc-updater`, `decision-lookup`) under `milestone-driven/agents/`, installed alongside skills by `install.sh`. `doc-updater` runs per task inside `/implementation-cycle`, keeping reference docs and examples in sync incrementally; `/milestone-closing` then handles the holistic README narrative at the end of the milestone. `decision-lookup` is a read-only librarian for the project's Architecture Decision Records (see "Decision records" below): given a topic, it reads the crisp per-topic summaries and the decision index (`architecture/decisions.md` by default, or under the `architecture-path:` directory set in `CLAUDE.md`), pulls only the relevant records, and returns a compact briefing — spawned by `/strategic-planning` and `/milestone-breakdown` so those opus sessions inherit prior decisions without paging the whole log into context.

**Decision records (ADRs) and the architecture home.** All architectural material lives under one **architecture home** — `architecture/` by default, overridable per project via an `architecture-path: <directory>` line in `CLAUDE.md` (honored across the milestone-driven skills, `/adr`, the domain-driven skills, and spec-sharpener; `<architecture-home>` below denotes the resolved directory). It holds three things: full ADRs at `<architecture-home>/decisions/NNNN-kebab-title.md` (context, decision, rationale, alternatives, consequences — the **single source of truth** for a decision's reasoning); a one-sentence-per-decision index at `<architecture-home>/decisions.md` (the abbreviated form agents read to know a decision exists); and crisp, derived per-topic **summaries** at `<architecture-home>/<topic>.md` (e.g. `tech-stack.md`, `error-handling.md`) — the short "what the rules are" digest an LLM reads *instead of* paging the whole log. The decision-making skills record substantial *on-the-way* decisions — ones that split the architecture, commit to a goal, or foreclose an expensive-to-reverse alternative — as ADRs: `/project-inception` records the foundational tech-shape/distribution decisions, `/strategic-planning` records directional decisions a milestone commits to, `/milestone-breakdown` records milestone-level architectural splits, and (domain-driven) `/architecture-foundation` records the whole architectural foundation (tech stack, persistence, integration, testing, cross-cutting) as a batch of ADRs. Recording is done inline by each skill from the bundled `references/decision-record.md`, whose `Scope:` field captures any binding to a specific artifact (frontend/backend/mobile/CLI), environment (prod/test), or bounded context. **The per-topic summaries are derived, never authored by hand:** after any ADR is recorded, the recording skill spawns the `architecture-summarizer` agent (in `common/`), which maps each new ADR to its topic(s) and rewrites the affected `<architecture-home>/<topic>.md` files (skills that record via `Skill(adr)` inherit this automatically). *Querying* is delegated to the `decision-lookup` subagent (which reads the summaries first, then the index) for context housekeeping. `/task-implementation` and `/milestone-closing` consume the log directly (reading the specific referenced records / `decisions.md`) rather than via the subagent — the former because it may run inside a worker subagent that cannot spawn one, the latter because it only needs the index. `spec-sharpener` (in `common/`) runs pre-implementation and therefore writes **no** ADRs — the sharpened spec text is the record; it treats an existing decision log as read-only input (its surveyor drops findings already settled by `Accepted` decisions). `/adr` (in `common/`) is the manual entry point to the same log: the user invokes it to record a decision from the current conversation that no skill captured on its own. (The research workflow's `research/DECISIONS.md` is a separate, content-level log and is intentionally *not* part of this ADR convention.)

**Research workflow** — a multi-phase system for building knowledge bases:
1. `/research-inception` → Creates project structure (INDEX.md, DECISIONS.md, glossary.md, topic stubs)
2. `/research-add-topic` → Adds a new topic (directory + chapter stubs) to an existing project
3. `/research-add-chapter` → Adds new chapter stubs to an existing topic directory
4. `/research-inquiry` → Adds RESEARCH directives (section outlines) to a chapter stub
5. `/research-inquiry-cycle` → Sequentially batches `research-inquiry-worker` subagents over all `stub` topics; workers run fully in parallel within a batch (one topic each)
6. `/research-investigation` → Writes content for one section, marks confidence levels; runs as a forked `research-investigation-worker` subagent (`context: fork`) that drives the web search-fetch-verify loop inline
7. `/research-investigation-cycle` → Sequentially batches `Skill(research-investigation)` invocations over all pending RESEARCH directives; forks run in parallel across distinct topic files within a batch, serial within a topic
8. `/research-audit-consistency` → Checks cross-topic contradictions; inserts AUDIT directives. Delegates CONFIDENCE-marker verification to the `confidence-verifier` subagent.
9. `/research-audit-coverage` → Checks gaps relative to the research plan; inserts AUDIT directives. Delegates CONFIDENCE-marker verification to the `confidence-verifier` subagent.
10. `/research-audit-quality` → Checks depth and sourcing adequacy; inserts AUDIT directives. Delegates CONFIDENCE-marker verification to `confidence-verifier`, and fans out per-topic depth/sourcing analysis to `quality-auditor` subagents in parallel.
11. `/research-audit-coherence` → Checks narrative flow; inserts AUDIT directives. Delegates CONFIDENCE-marker verification to `confidence-verifier`, and fans out per-topic flow analysis to `coherence-auditor` subagents in parallel.
12. `/research-audit-topic` → Per-item counterpart to the four lens skills above: audits **one topic across every lens** (consistency, coverage, quality, coherence, graphics) in a single pass and advances it `draft → audited`. Runs as a forked `research-audit-worker` subagent (`context: fork`) that inlines all lens analysis and CONFIDENCE-marker verification — no nested subagents. Used directly, or as the per-topic unit fanned out by `/research-audit-cycle`.
13. `/research-audit-cycle` → Sequentially batches `Skill(research-audit-topic)` invocations over all `draft` topics; forks run in parallel across distinct topics within a batch, and each topic's lenses run serially inside its fork. Takes a max-items cap and a parallelization factor (`[max-items][@workers]`), drives topics to `audited`, and is resumable/idempotent (only picks up remaining `draft` topics).
14. `/research-refine` → Resolves AUDIT findings (correct, expand, condense, restructure, etc.)
15. `/research-refine-cycle` → Sequentially batches `research-refine-worker` subagents over the project's open AUDIT directives; one worker per topic file (each resolves that file's AUDITs serially via `research-refine`), workers run in parallel across distinct files/directories within a batch. Takes a `<count|all>@<workers>` argument (how many AUDIT directives to resolve, how many workers per batch) and is resumable/idempotent (re-derives the open-AUDIT set each pass, so it only picks up what still needs resolving). Ships a `list-open-audits.sh` helper that deterministically enumerates every open AUDIT directive ordered by INDEX.md.
16. `/research-restructure` → Structural changes (split, merge, promote, demote, nest, flatten) at any depth in the topic tree, with cross-reference rewriting
17. `/research-glossary-sync` → Reconciles glossary.md against topic content. Fans out per-topic candidate extraction to `term-extractor` subagents in parallel.
18. `/research-ingest-source` → Out-of-band author entry point (user-invoked, not part of the phase cycle): given one specific source the author already has (URL or local file), vets it for legitimacy with the same levers as investigation (URL verification, primary-vs-secondary, independence, per-claim confidence), then weaves it into every existing section it corroborates/extends/contradicts across the topic tree — reusing the reference, CONFIDENCE, contradiction (both-positions + DEC + AUDIT), and gap-AUDIT conventions. This is the source-first counterpart to `/research-investigation` (which is directive-first and discovers its own sources). Delegates the read-heavy placement scan to the `corpus-locator` subagent and confirms the placement plan before editing.
19. `/research-status` → Read-only progress report. Runs the shared `research-status.sh` helper to derive and print each chapter's status live from on-disk signals, with detail counts of what is still missing. The human-facing front end to the helper every other research skill consults for status.

**Derived status, not a stored enum.** Chapter status (`stub → inquiry → draft → audited → done`) is **never stored** — no `**Status**:` line in `INDEX.md`, no status frontmatter field. It is derived on demand as the **earliest phase with an open worklist marker** — `RESEARCH ≺ CONFIDENCE ≺ AUDIT` — where each marker type has exactly one producer and one consumer (inquiry→investigation→audit→refine), with the frontmatter `audit:` field (which core lenses have run) disambiguating a clean draft from `done`. The audit phase is **total over CONFIDENCE**: it verifies-and-removes every marker or converts it to an AUDIT, so no CONFIDENCE marker survives audit and CONFIDENCE never independently gates `done` (a stray one at four lenses regresses the chapter to `draft`). An unverified `references.yaml` entry is likewise folded into the marker pipeline — a CONFIDENCE marker pre-audit, an AUDIT post-audit — not a separate gate. `done` therefore reduces to "four core lenses recorded and no open RESEARCH/CONFIDENCE/AUDIT marker". The single source of truth is the shared `research-status.sh` helper (bundled in the `research-status` skill directory; invoke as `bash <skills-root>/research-status/research-status.sh <research-dir> [--path P] [--status S]`). Every skill that used to read or write an `INDEX.md` status now reads the derived status from this helper instead and writes no status anywhere — which also removes the old cross-cutting "status reconciliation" / race-recovery steps from the cycle skills. `INDEX.md` is purely the outline plus abstracts. Full derivation rules are in `research/README.md`.

The research workflow ships nine custom subagents under `research/agents/`, installed alongside skills by `install.sh`:
- `research-inquiry-worker` — per-topic inquiry worker (invokes `research-inquiry`) spawned in parallel batches by `research-inquiry-cycle`.
- `research-refine-worker` — per-file refine worker; resolves one topic file's open AUDIT directives serially (invoking `research-refine` once per directive) and returns a report block. Spawned in parallel batches by `research-refine-cycle` (one worker per file).
- `research-investigation-worker` — execution environment for the forked `research-investigation` skill (`context: fork`). Hosts one directive per fork, including the inline web search-fetch-verify loop. Spawned in parallel batches by `research-investigation-cycle` and also used by direct human invocations of `/research-investigation`.
- `research-audit-worker` — execution environment for the forked `research-audit-topic` skill (`context: fork`). Hosts one topic per fork and runs every audit lens (consistency, coverage, quality, coherence, graphics) plus CONFIDENCE-marker verification inline — no nested subagents. Spawned in parallel batches by `research-audit-cycle` and also used by direct human invocations of `/research-audit-topic`.
- `confidence-verifier` — shared CONFIDENCE-marker verifier for the four standalone `research-audit-*` lens skills (the forked `research-audit-topic` resolves markers inline instead).
- `quality-auditor` — per-topic depth/sourcing auditor spawned in parallel by `research-audit-quality`.
- `coherence-auditor` — per-topic narrative-flow auditor spawned in parallel by `research-audit-coherence`.
- `term-extractor` — per-topic glossary-candidate extractor spawned in parallel by `research-glossary-sync`.
- `corpus-locator` — read-only placement scout spawned by `research-ingest-source`. Given a new source's extracted claims, scans the topic tree and returns, per claim, the sections that corroborate/extend/contradict it (plus uncovered claims). Does not fetch the web or edit files.

The full research specification is in `research/README.md`.

**Codebase-survey workflow** — bootstraps and maintains an AI-consumable map of an existing codebase, with detail co-located alongside code:
1. `/codebase-survey-init` → Bootstrap; delegates raw discovery to the structural-discovery subagent, synthesizes the module map in the main session, writes top-level `CODEBASE.md` plus per-module stubs
2. `/codebase-survey-module <path>` → Per-module deep-dive; spawns five subagents (dep-grapher, api-surface-extractor, wire-api-extractor, test-auditor, ops-detective) in parallel and assembles `<path>/CODEBASE.md`
3. `/codebase-architecture-assessment` → Cross-cutting pass; writes all four `docs/codebase/*.md` files — `assessment.md` (findings tagged `kind: rule` or `kind: observation`), plus synthesised `architecture.md`, `tech-stack.md`, and `operations.md`. Files that lack the `generated_by` marker are treated as human-authored stated architecture and not overwritten.
4. `/codebase-survey-update [commit-range|PR#]` → Incremental refresh driven by per-module `surveyed_sha` deltas; flags `CLAUDE.md` drift but does not rewrite
5. `/codebase-derive-instructions` → Lifts `kind: rule` findings into `CLAUDE.md` (or `AGENTS.md` if present) with source-anchor comments; verifies length, imports, and rule count before writing

The subagents live in `codebase-survey/agents/` and are installed alongside skills by `install.sh`.

**Synaptic-authoring workflow** — skills for authoring content for **Synaptic**, an interactive online learning platform built in parallel by the user. A Synaptic *track* is a git directory of content files validated and snapshotted by the deterministic `synaptic` CLI; these skills draft that content against the CLI's file contract and defer every integrity judgment to it — they never adjudicate validity, mint ids, or hash anything themselves (ids are minted only by `synaptic scaffold`). The content contract, DAG invariants, and `grounding` metadata block the skills target are specified in `synaptic-authoring/README.md`:
1. `/author-ingest` → Distils repo-local source material (a citable research KB or plain repo docs — never the web) into un-`id`'d `reference/` files tagged with resolvable grounding refs, and records an ingest-state manifest (`reference/.ingest-state.yaml`: source root, watermark SHA, per-file provenance) for later incremental updates. Delegates fact extraction to the `material-extractor` subagent — one call for a small corpus, fanned out per file/subtree in parallel (with an in-session merge) for a large one
2. `/author-ingest-update` → Delta-aware re-ingest: given a git commit range (or defaulting to the ingest-state watermark), diffs the ingested source, re-extracts **only** the changed files (reusing `material-extractor`), reconciles them against the existing `reference/` layer by grounding ref (added/changed/anchor-moved/removed), and — closing the pipeline's silent-drift gap — delegates the track-wide node scan to the `grounding-tracer` subagent to report which nodes are STALE (content moved under a stable ref → needs `/author-snippet`) or BROKEN (ref no longer resolves → needs re-grounding). Confirms the plan before writing, bumps the watermark, and touches only `reference/` + the manifest (never node files). Repo-local only
3. `/author-structure` → Proposes the track's prerequisite DAG (node list, edges, priority) from the `reference/` corpus + a track goal, then mints node ids via `synaptic scaffold` once the human approves. Delegates DAG proposal to the `concept-mapper` subagent
4. `/author-snippet` → Drafts the learner-facing body of a scaffolded node from `reference/` material — playful low-stakes voice, always stating *why it matters* / *what it unlocks*, each claim carrying a resolvable grounding ref; proposes the glossary/cheatsheet slug links the prose needs. Delegates the read-heavy corpus/slug sweep and the draft to the `snippet-drafter` subagent, then scaffolds slugs and writes the node
5. `/author-snippet-cycle` → Batch counterpart to `/author-snippet`: drives every un-drafted scaffolded node to a drafted body by fanning `snippet-drafter` across nodes in parallel batches, then **centrally** scaffolding the union of slugs they need (dedup by slug, so parallel drafters never race to create the same term) and writing each node. Takes a `[max-nodes][@workers]` argument and is resumable/idempotent (each pass re-derives the un-drafted set from `TODO` placeholders)
6. `/author-questions` → Drafts multiple-choice questions with tight reference lists honoring "assessment is feedback, never a gate", then mints question ids via `synaptic scaffold` and writes the files. Delegates question drafting to the `question-smith` subagent
7. `/author-questions-cycle` → Batch counterpart to `/author-questions`: drives every drafted-but-unquestioned node to having assessment by fanning `question-smith` across nodes in parallel batches, then **centrally** minting question ids and writing files. Takes a `[max-nodes][@workers]` argument and is resumable/idempotent (each pass re-derives the unquestioned set)
8. `/author-gap-scan` → Audits an existing or proposed DAG for foundational gaps (concepts referenced but never taught, orphan roots, prerequisite leaps, redundant nodes) and produces a prioritized gap report + authoring plan; read-only. Delegates to the `concept-mapper` and `coverage-auditor` subagents
9. `/author-selfcheck` → The standing hand-off gate: runs `synaptic validate --json` over the track, summarizes findings, and refuses to present an integrity-breaking snapshot to a human

The synaptic-authoring workflow ships six custom read-only proposal subagents under `synaptic-authoring/agents/`, installed alongside skills by `install.sh`. Each carries `tools: Read, Glob, Grep`, returns a structured report, and writes no files — the orchestrating skill does all scaffolding and writing:
- `material-extractor` — pulls atomic knowledge units from source files (each with candidate prerequisites, a "why it matters" hook, and a source line), reports which grounding kind applies (research vs document), **and returns the write-ready `reference/` file body per source file** so the orchestrator writes it verbatim and never re-reads the source into its own context (the double-read that otherwise blows the token budget). For `/author-ingest` (one call for a small corpus, fanned out per file/subtree for a large one) and `/author-ingest-update` (which runs it over only the changed source files).
- `concept-mapper` — turns knowledge units into a proposed prerequisite DAG with a one-line rationale per edge and an explicit acyclicity self-check. For `/author-structure` and `/author-gap-scan`.
- `snippet-drafter` — locates the `reference/` unit(s) a scaffolded node should ground on (lifting confidence forward), finds the glossary/cheatsheet slugs the prose needs (existing vs to-scaffold), and drafts the learner-facing title + body — returned as a proposal, writing nothing. For `/author-snippet` and, fanned out in parallel batches, `/author-snippet-cycle`.
- `question-smith` — drafts multiple-choice questions with per-distractor rationale and flags any question whose reference list is too broad. For `/author-questions` and, fanned out in parallel batches, `/author-questions-cycle`.
- `coverage-auditor` — audits a track's DAG for foundational gaps keyed to node ids. For `/author-gap-scan`.
- `grounding-tracer` — read-only downstream-impact scout: given the grounding refs a source update changed (tagged changed / anchor-moved / removed) and the track root, scans the id'd node files and returns a STALE/BROKEN worklist (plus each affected node's dependents). Keeps the track-wide grep sweep out of the orchestrator's context. For `/author-ingest-update`.

**Domain-driven workflow** — a build workflow that takes a project from a blank page to shipped code via Domain-Driven Design strategic modeling, then drives a dependency-ordered task backlog to completion. It favors *flow over batches*: there is **no milestone layer** — a growing backlog whose scheduling is governed by an explicit dependency DAG, organized primarily by bounded context. Full spec in `domain-driven/README.md`:
1. `/grounding` → Socratic vision session (adapted from the Agentheim brainstorm skill); produces a single tight `vision.md` at the project root and **stops there** — every later phase is deliberately separate so the human keeps control.
2. `/domain-model` → big-picture EventStorming: a chronological domain-event timeline (marking events that originate outside the software and the situation that triggers each), commands/actors for the in-software events, policies, external systems, the aggregates that own consistency, and a hotspots list of unresolved decisions → `domain-model.md`. Seeds the first pass with the `domain-seed-extractor` subagent, refines Socratically, and offers to turn hotspots into ADRs (via `common/adr`). Re-entrant: with `domain-model.md` present it runs in **revision mode** — diff-oriented, fed by the `deviated` drift worklist (see below) — and hands off to `/context-mapping` when aggregate clusters change.
3. `/context-mapping` → draws the bounded contexts around the model's aggregate clusters and records their relationships (partnership, customer/supplier, conformist, ACL, published language, shared kernel) plus each context's ubiquitous language → `context-map.md` + `bounded-contexts/<context>.md`. Seeds with the `boundary-proposer` subagent; offers boundary/relationship decisions as ADRs. The context map is the **domain-compliance referent** every task is checked against. Re-entrant: with `context-map.md` present it runs in **revision mode** (same drift worklist), closing with a **backlog ripple pass** — live tasks under a renamed context are mechanically re-slugged, tasks under a split/merged one flagged for `/task-refine` to re-home.
4. `/architecture-foundation` → Socratic session (requires vision + domain-model + context-map present) that defines the project's architectural boundaries and guidelines. Opens its agenda with the domain model's still-open **hotspots** — the last gate before the build phase: each is resolved (→ ADR, ticked off in `domain-model.md`), explicitly parked by the human, or reclassified as non-architectural, so nothing unresolved silently survives into `/task-append`. Then works general → specific: tech stack (languages/frameworks/runtimes), persistence/data stores, communication & integration between contexts, testing principles, and cross-cutting concerns (error handling, observability, security, configuration, versioning, deployment). Seeds an agenda with the `architecture-proposer` subagent; records each settled decision as an ADR (via `common/adr`, which keeps the derived `architecture/<topic>.md` summaries in sync), making artifact/environment/bounded-context binding explicit. Closes a first run by proposing a **walking-skeleton task** — one thin vertical slice exercising the stack, one persistence decision, and one context-map relationship end-to-end (handed to `/task-append` on approval) — so wrong ADRs surface while superseding them is cheap. Re-entrant: with ADRs but no landed code it *extends* the foundation; once tasks have landed it runs in **revision mode** — reading the architecturally-implicated `deviated` tasks' closing records and superseding the decisions reality contradicted (clearing the flags it fully handles). The resulting guidelines are the **architecture-compliance referent** every task is checked against.
5. `/task-append` → captures one or more tasks into the backlog as `draft`s from human input of any quality (polished spec or raw brain-dump), splitting on `---` document delimiters so several tasks can be captured at once; mints an id per task via `tasks.sh`, writes `tasks/NNNN-slug.md`, does **not** interview. Human-serial.
6. `/task-refine` → turns a `draft` into a ready `todo`: assesses completeness, domain-compliance (against the target context's ubiquitous language + relationships), and size — delegated to the `task-analyzer` subagent — interviews the human, **splits** oversized tasks (children minted as `todo`, original becomes an inert `split` tombstone with `split_into`, dependents rewired to the correct children in a separate pass), wires `depends_on`, and attaches `related_adrs`. Runs `tasks.sh check-dag` as a hard gate after wiring. Human-serial.
7. `/task-cycle` → drives ready `todo` tasks (from `tasks.sh ready`) to `done` via `task-worker` subagents (strict TDD → verify → `Skill(commit)`). Takes `[<limit>|all][@<workers>]` (default `all@1`). `@1` works in-place sequentially; `@N` implements in parallel git worktrees and delegates sequential merge-back to the `integrator` subagent (bounce-on-conflict). **The orchestrator owns every status write** (and every task-file write) — workers never touch `status` or the file; each worker returns its human-verification (`## Manual testing`) and `## Deviations from plan` records in its report, and the orchestrator writes them into the task body when the task lands — setting `deviated: true` in the frontmatter when the deviations are non-trivial, which feeds the drift worklist the revision runs consume. Resumable/idempotent (re-derives the ready-set each pass).
8. `/task-status` → read-only backlog board; the human front end to the bundled `tasks.sh` query helper.
9. `/whats-next` → the forward-looking companion to `/task-status`: assesses `vision.md`, `domain-model.md`, and `context-map/` against the current backlog state (read through `tasks.sh`, frontmatter only — never scanning task bodies), surfaces coverage gaps (uncovered aggregates, thin contexts, unrepresented vision outcomes, blocking hotspots, an **unvalidated foundation** — architecture ADRs recorded but no landed task exercises them end-to-end, making a walking-skeleton vertical slice the top proposal — and **drift** — an accumulating `deviated` worklist, dangling context slugs; when drift dominates it recommends a revision instead of more tasks: `/domain-model` or `/context-mapping` for model drift, `/architecture-foundation` when the deviated tasks' `related_adrs` hint the friction is architectural), and proposes a prioritized list of next tasks. Read-mostly and **advisory** — it hands approved suggestions to `/task-append` (the single human-serial capture point) and mints ids / wires dependencies / refines nothing itself. Runs in the main session with no dedicated subagent (like `/task-status`).

**Derived scheduling, stored status.** Unlike research (derived status), a task's `status` is a *stored* enum — its transitions are human/orchestrator decisions, not content-derivable signals. But the **scheduling** is derived deterministically: the backlog can grow large, so **no skill or subagent ever scans the task corpus** — every question (`ready`, `next-id`, `by-status`, `by-context`, `deviated`, `get`, `blockers`, `dependents`, `check-dag`, `board`) is answered by the `tasks.sh` helper (bundled in the `task-status` skill dir; invoke as `bash <skills-root>/task-status/tasks.sh <cmd> [--dir tasks]`), which parses only YAML frontmatter (`yj -yj` → `jq`) and returns ids/counts. Frontmatter is the query index; prose bodies are read only by the one worker implementing that one task — with one sanctioned exception: a `/domain-model`, `/context-mapping`, or `/architecture-foundation` revision reads the `## Closing` sections of exactly the ids `tasks.sh deviated` lists (a bounded, flag-gated worklist, not a scan; each revision clears the flags it fully handles). A task's canonical id is the `NNNN-` **filename** prefix (dodges YAML octal parsing); `depends_on`/`split_into` are normalized to 4-digit strings. Requires `yj` and `jq`.

The domain-driven workflow ships six custom subagents under `domain-driven/agents/`. Four are read-only proposal scouts (`Read, Glob, Grep`; write nothing): `domain-seed-extractor` (first-pass EventStorm from `vision.md`, for `/domain-model`), `boundary-proposer` (first-pass context map + relationships from the domain model, for `/context-mapping`), `architecture-proposer` (first-pass architecture agenda — open hotspots first, then tech stack, persistence, integration, testing, cross-cutting — from vision + domain model + context map, for `/architecture-foundation`), and `task-analyzer` (per-task refine assessment — completeness/domain-compliance/size/deps/decisions — for `/task-refine`). Two are write-side workers: `task-worker` (TDD-implements one claimed task and commits it via `Skill(commit)`; never edits `status`) and `integrator` (sequential worktree merge-back, bounce-on-conflict). It **depends on `common` being installed alongside it** for `/adr` (records hotspot/boundary/foundation/task decisions, and spawns `architecture-summarizer` to keep the topic summaries current) and `/commit` (the single commit point); `/task-refine` reads the architecture guidelines directly (via `task-analyzer`) rather than via `decision-lookup`, keeping it self-contained. (This workflow's ADRs follow the shared architecture-home convention above — `architecture/` by default, overridable via `architecture-path:` in `CLAUDE.md`.)

**Common workflow** — skills and agents that aren't owned by any single workflow live under `common/`. This currently houses two kinds of skill:

Cross-workflow tools (used by, or invoked from, multiple workflow families):
- `/commit` → Crafts a conventional commit from staged/unstaged changes; the single commit point for all workflows (milestone-driven, research, codebase-survey) — no other skill commits directly
- `/pr` → Creates or updates a GitHub pull request for the current branch via `gh`. Synthesises a What/Why/How body from commits and diff, defaults to draft (override with `final`), auto-pushes the branch, and refuses on a dirty tree (defers to `/commit`)
- `/version-bump` → Bumps the project's declared version (detects `package.json`, `Cargo.toml`, `pyproject.toml`, and similar manifests) from an argument that is either an explicit `x.y.z` or one of `major`/`minor`/`patch` (with `major`/`minor` truncating lower components to zero), then cuts an annotated git tag. Gathers a changelog from the commits since the last version tag via the `changelog-gatherer` subagent and writes it to three destinations: `CHANGELOG.md`, the tag annotation, and the chat. Reads (tags, manifests, CHANGELOG) are inlined into the skill via `!`-command preprocessing. Defers the release commit to `/commit` (the single commit point) and tags the resulting commit; never pushes without being asked

Standalone utilities (don't belong to any workflow):
- `/setup-github-workflow` → User-invoked only. Analyzes the current project (stack signals from build manifests/lockfiles, distribution shape from the README) and proposes GitHub Actions workflows for CI and releases tailored to the technology and goal; when workflows already exist, refreshes every action/library to the latest available version and re-pins to the current commit SHA. Interviews the human on the judgement calls (branching model — trunk-based / PR-based / release-branch —, when CI runs, when a release is cut and what it produces, which secrets are needed) until there is a shared understanding. **Always pins actions to a specific commit SHA** at the most up-to-date version available at call time, with a `# vX.Y.Z` comment. Delegates version resolution and security assessment of target versions to the `action-security-auditor` subagent and surfaces every finding to the human. Asks for explicit confirmation before writing; never commits or pushes (defers commit to `/commit`). Repository reads (existing workflows, manifests, README, git remote/branch/tags, CHANGELOG) are inlined into the skill via `!`-command preprocessing.
- `/adr` → Manually records one or more ADRs from the current conversation under the project's architecture home (`architecture/decisions/` by default, overridable via `architecture-path:` in `CLAUDE.md`; shared `NNNN-title.md` + `decisions.md` convention — see "Decision records" above), then spawns `architecture-summarizer` to refresh the derived `architecture/<topic>.md` summaries. The human override for when a decision worth preserving was made in-session but no skill recorded it; model-invocable (triggers when the user asks to record/capture/log a decision), treats the invocation itself as the worth-recording call, grounds every section in what was actually discussed (never fabricates alternatives), and confirms the decision list before writing
- `/audit-context` → Diagnoses contradictions, ambiguities, and irrelevance in the current session context (or a given file list); read-only, produces a line-cited severity-ranked report
- `/deckset` → Generates Deckset (macOS) presentations from existing markdown content
- `/spec-sharpener` → Hardens a greenfield project's spec/docs into an implementation-ready state; interviews the user one issue at a time (ambiguities, contradictions, gaps) and edits the docs in place — the sharpened spec itself is the record. Runs pre-implementation, so it writes no ADRs; an existing decision log (the architecture home) is read-only input for deduplication (see "Decision records" above). Keeps the main session lean by delegating both the doc-heavy sweep and the file-writing to subagents (see below) — the main session holds only the compact backlog and runs the interview

The common workflow ships five custom subagents under `common/agents/`, installed alongside skills by `install.sh`. The first two are used by `/spec-sharpener`, the third by `/version-bump`, the fourth by `/setup-github-workflow`, the fifth by `/adr` (and every other ADR-recording skill):
- `spec-surveyor` — read-only reconnaissance worker. Discovers the docs, reads the decision log, builds a system model, sweeps against the finding taxonomy, drops findings already settled by `Accepted` decisions, and returns a compact prioritized backlog (each finding carrying a quoted anchor, the problem, why-it-matters, and 2–4 concrete options). All the doc text and the taxonomy stay inside the subagent and are discarded.
- `decision-encoder` — write-side worker. Given one resolved finding plus the agreed resolution, makes the minimal edits to the affected docs so they state the resolution unambiguously; writes no ADR or index entry. Returns a one-line confirmation. Runs one at a time (never in parallel) since successive resolutions may touch the same document.
- `changelog-gatherer` — read-only changelog worker. Given a commit range (or the whole history) and the new version string, runs `git log` over that range, classifies commits into Keep-a-Changelog sections, filters noise (merges, release commits, internal churn), and returns a single ready-to-use changelog block. Writes no files, commits, or tags — `/version-bump` writes the block to CHANGELOG.md, the tag message, and the chat. The raw commit list stays inside the subagent.
- `action-security-auditor` — version-resolution and security worker. Given the list of GitHub Actions (`owner/repo`) `/setup-github-workflow` intends to use (plus current pins on a refresh), resolves each to its latest stable release, pins it to the exact **commit SHA** for that release (via `gh api`/`git ls-remote`, dereferencing annotated tags to the commit), and assesses the target version for security risk (known CVEs/GHSAs, recently-compromised tags, maintainer/ownership changes, unpinned transitive references). Returns per-action version tag + full SHA + risk level + actionable notes; writes no files. The orchestrating skill surfaces every non-`none` finding to the human before that action lands in a workflow. Registry/git/web lookups stay inside the subagent.
- `architecture-summarizer` — write-side worker. Given the architecture home and the number(s) of the just-recorded ADR(s), reads those records, maps each to one or more architectural topics (tech-stack, data-persistence, api-and-integration, testing, error-handling, observability, security-auth, configuration, build-deploy, code-style, …), and rewrites the affected `<home>/<topic>.md` summary files as crisp, current guidelines that link back to the ADRs (making artifact/environment/context scope explicit). May create/split/merge topic files. Never touches the ADRs or `decisions.md` — the summaries are a derived digest. Spawned by `/adr` and by the milestone-driven inline recorders (`/project-inception`, `/strategic-planning`, `/milestone-breakdown`); skills recording via `Skill(adr)` inherit it.

### Skill File Conventions

Each `SKILL.md` has YAML frontmatter controlling behavior:
- `name` — becomes the `/slash-command`
- `description` — helps Claude decide when to auto-load the skill
- `model` — which Claude model to use (opus for planning/research, sonnet for implementation/commit)
- `disable-model-invocation: true` — user-only invocation (all research skills use this)
- `argument-hint` — documents expected arguments

Reference files (like `milestone-driven/skills/milestone-breakdown/references/SAMPLE-PLAN.md`) sit alongside SKILL.md and are loaded as context.

### When Adding a New Skill

After creating a new skill under `<workflow>/skills/<skill-name>/`, register it in both:
- `CLAUDE.md` — add it to the appropriate workflow family list under "Workflow Families" (or create a new family if it doesn't fit)
- `README.md` — add it to the user-facing skill listing

If the skill doesn't belong to any existing workflow, place it under `common/skills/` and consider whether a new workflow directory is warranted.

Keep the one-line description consistent across both files.

If a **new workflow directory** is added (not just a new skill in an existing one), also update the marketplace metadata:
- Create `<workflow>/.claude-plugin/plugin.json` (name, description, author, homepage, repository, keywords) — copy the shape from an existing workflow's manifest.
- Add a new plugin entry to `.claude-plugin/marketplace.json` (`name`, `source: "./<workflow>"`, `description`, `category`).
- Keep the plugin `description` in sync with the workflow's one-line summary in CLAUDE.md and README.md.

Adding a skill *within* an existing workflow does not require touching `.claude-plugin/` — the workflow plugin already picks up everything under its `skills/` directory.

### Subagents

Custom subagents live in `<workflow>/agents/<name>.md` (a single file per agent, not a directory) and are installed via symlink to `~/.claude/agents/` (or `<project>/.claude/agents/` for project installs) by the same `install.sh`. Skills invoke them via the `Agent` tool with `subagent_type: <name>`. Use a custom subagent when a skill needs to delegate a deterministic, structured task (e.g., extracting an API surface, running a dependency grapher) so the orchestrating skill never sees raw tool output.

### Key Documents Referenced by Skills

Skills expect these files to exist in target projects:
- `README.md` — project identity (created by project-inception)
- `ROADMAP.md` — milestone **index**: one line per milestone (`NNNN-slug.md — [status] summary`), kept lean so it stays cheap to load. Full milestone content (value, outcome, success criteria, notes, closing notes) lives in `roadmap/NNNN-slug.md`. Managed by strategic-planning, read by milestone-breakdown/closing (which open only the specific milestone file they need)
- `PLAN.md` — task list for current milestone (managed by milestone-breakdown, consumed by task-implementation)
- `<architecture-home>/decisions/NNNN-kebab-title.md` (full ADRs) + `<architecture-home>/decisions.md` (index) + `<architecture-home>/<topic>.md` (crisp derived per-topic guideline summaries) — the architecture home, where `<architecture-home>` is `architecture/` by default or the `architecture-path:` directory set in `CLAUDE.md` (ADRs written by project-inception, strategic-planning, milestone-breakdown, and domain-driven's architecture-foundation; summaries derived by the `architecture-summarizer` agent after each ADR; read via the `decision-lookup` subagent by strategic-planning/milestone-breakdown, directly by task-implementation/milestone-closing, and as read-only dedup input by spec-sharpener's surveyor)

Research skills expect an `research/` directory with `INDEX.md`, `DECISIONS.md`, `glossary.md`, and `content/` subdirectory. Chapter status is derived (not stored) — see the "Derived status, not a stored enum" note under the Research workflow above.

Codebase-survey skills produce and consume:
- `CODEBASE.md` at the repo root (top-level survey, module map, tech stack)
- `<module>/CODEBASE.md` per module (purpose, FRs/NFRs, deps, API, tests, deviations, ops)
- `docs/codebase/architecture.md`, `tech-stack.md`, `operations.md`, `assessment.md`
- All survey files carry front-matter: `surveyed_sha`, `surveyed_at`, `survey_schema`. Derived `CLAUDE.md` carries `derived_from_survey_sha`, `derived_at`, `derive_schema`.
